# Javadoc Extraction Rules and API Reference Formatting

This reference defines how the JavaFX DocGen skill parses Java source files to extract Javadoc comments and formats them into a structured API reference document. It covers comment discovery, tag extraction, package hierarchy organization, Markdown rendering, and handling of undocumented elements.

## Overview

The API reference is generated by scanning every `.java` file under `src/main/java/`. Only `public` and `protected` declarations are included in the reference; `package-private` and `private` members are omitted unless explicitly requested. The output is organized by package, then by class, and finally by member (constructors, methods, fields).

## Javadoc Comment Structure

A Javadoc comment begins with `/**` and ends with `*/`. It must appear immediately before the declaration it documents (no blank lines in between). The comment has two parts:

1. **Description** — free-form text from the opening `/**` up to the first block tag.
2. **Block tags** — lines starting with `@` (`@param`, `@return`, `@throws`, `@see`, `@author`, `@version`, `@since`, `@deprecated`).

```java
/**
 * Manages persistent application preferences for the JavaFX desktop client.
 * <p>Preferences are stored in the user's home directory and loaded on startup.
 *
 * @author DongZY0617
 * @version 1.2
 * @since 1.0
 */
public class PreferenceManager { ... }
```

## Parsing Java Source Files

Source files are parsed with a lightweight line-based scanner. The scanner tracks the current declaration context (class, method, field) and associates each Javadoc block with the declaration that follows it.

```text
Algorithm:
  1. Read file line by line.
  2. When a line matches /^\s*\/\*\*/ , start capturing a Javadoc block.
  3. Continue capturing until a line matching /\*\// is found.
  4. The next non-blank, non-comment line is the documented declaration.
  5. Classify the declaration as class | interface | enum | method | field.
  6. Attach the captured Javadoc block to the declaration object.
```

If no Javadoc block precedes a declaration, the member is flagged as undocumented.

## Class-Level Documentation

For each public class, interface, or enum, extract:

- **Description** — first paragraph of the class Javadoc.
- `@author` — author name(s); multiple authors become a comma-separated list.
- `@version` — version string.
- `@since` — version since which the class exists.
- `@deprecated` — marks the class as deprecated.

```java
/**
 * Service for exporting tabular data to CSV files.
 *
 * @author DongZY0617
 * @version 2.0
 * @since 1.1
 */
public class CsvExporter implements DataExporter { ... }
```

## Method-Level Documentation

For each public or protected method (including constructors), extract:

- **Description** — method purpose.
- `@param <name> <description>` — one entry per parameter.
- `@return <description>` — return value description (omit for `void`).
- `@throws <Exception> <description>` — one entry per declared exception.
- `@since`, `@deprecated` — as applicable.

```java
/**
 * Exports the given rows to a CSV file using the configured delimiter.
 *
 * @param rows    the data rows to export; must not be null
 * @param target  the destination file path
 * @return the number of rows written
 * @throws IOException              if the file cannot be written
 * @throws IllegalArgumentException if {@code rows} is null
 */
public int export(List<Row> rows, Path target) throws IOException { ... }
```

## Field-Level Documentation

For each public or protected field (including enum constants), extract the description and any `@see` references. Constants typically carry a short description explaining the value's meaning.

```java
/** Default delimiter used when none is specified. */
public static final String DEFAULT_DELIMITER = ",";
```

## Tag Extraction Rules

| Tag | Scope | Extraction Rule |
|-----|-------|-----------------|
| `@author` | class | Collect all occurrences; join with `, ` |
| `@version` | class | Take the last occurrence |
| `@param` | method | One entry per parameter; key = param name |
| `@return` | method | Single block; omitted if method returns `void` |
| `@throws` / `@exception` | method | One entry per exception type |
| `@see` | any | Collect all; used for cross-references |
| `@since` | any | Single version string |
| `@deprecated` | any | Boolean flag plus optional explanation |

Inline tags (`{@link ...}`, `{@code ...}`) within descriptions are converted to Markdown: `{@link Foo}` becomes ``[`Foo`](#foo)``, and `{@code text}` becomes `` `text` ``.

## Package Hierarchy Organization

Classes are grouped by their declared package. The output document mirrors the package tree:

```markdown
## Package `com.example.app.model`
### Class `User`
...
### Class `Order`
...

## Package `com.example.app.service`
### Class `UserService`
...
```

Each package section opens with a one-line overview listing the contained classes and their category (class, interface, enum, record).

## Markdown Formatting for API Reference

Each class reference follows this template:

```markdown
### Class `CsvExporter`

`public class CsvExporter implements DataExporter`

Exports tabular data to CSV files.

- **Author**: DongZY0617
- **Version**: 2.0
- **Since**: 1.1

#### Constructors

| Signature | Description |
|-----------|-------------|
| `CsvExporter()` | Creates an exporter with the default delimiter. |

#### Methods

| Method | Description |
|--------|-------------|
| `int export(List<Row>, Path)` | Exports rows to a CSV file. |

#### Fields

| Field | Type | Description |
|-------|------|-------------|
| `DEFAULT_DELIMITER` | `String` | Default delimiter used when none is specified. |
```

## Handling Missing Javadoc

When a public member has no Javadoc comment, the generator inserts a placeholder rather than leaving a blank entry. This keeps the reference complete and surfaces documentation gaps in the coverage report.

```markdown
| `void clearCache()` | *No documentation available.* |
```

The coverage metric (`coverage.api.coverage_percent`) counts these placeholders. A member marked *No documentation available* counts as undocumented; any member with a non-empty description counts as documented.

## Cross-Referencing Between Classes

`@see` tags and `{@link}` inline tags create cross-references. The generator resolves the target class name to an anchor in the same document:

```java
/**
 * @see CsvExporter
 * @see DataExporter#export(List, Path)
 * {@link PreferenceManager Preference settings}
 */
```

Rendered output:

```markdown
- See also: [CsvExporter](#csvexporter), [DataExporter.export](#dataexporter)
- Related: [Preference settings](#preferencemanager)
```

Unresolvable references (target class not in the project) are rendered as plain text and flagged with a warning in the docgen report. Anchor names are derived by lowercasing the simple class name; method references append a hash fragment of the method signature.

## Javadoc HTML Generation

In addition to the Markdown API reference, DocGen can generate standard Javadoc HTML output using the JDK's built-in `javadoc` tool. This produces the familiar framed Javadoc site with full cross-linking, search, and syntax-highlighted source views.

### When to Generate HTML

HTML generation is controlled by `.loop-config.json`:

```json
{
  "doc_javadoc_html": true
}
```

| Setting | Behavior |
|---------|----------|
| `true` | Generate both Markdown (`docs/api-reference.md`) and HTML (`docs/api-reference-html/`) |
| `false` (default) | Generate Markdown only |
| Not set | Default to `false` (Markdown only) |

### Maven Javadoc Plugin Configuration

Generate HTML Javadoc using the Maven Javadoc plugin:

```bash
mvn javadoc:javadoc -DdestDir=api-reference-html -DoutputDirectory=docs
```

Or with explicit options for better output:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-javadoc-plugin</artifactId>
  <version>3.6.3</version>
  <configuration>
    <outputDirectory>${project.basedir}/docs</outputDirectory>
    <destDir>api-reference-html</destDir>
    <show>public</show>
    <nohelp>true</nohelp>
    <doclint>none</doclint>
    <additionalOptions>
      <additionalOption>-html5</additionalOption>
      <additionalOption>--allow-script-in-comments</additionalOption>
    </additionalOptions>
  </configuration>
</plugin>
```

### JDK 17+ Module-Aware Javadoc

For modular JavaFX projects (`module-info.java` present), the `javadoc` tool automatically resolves module dependencies. Ensure the module path includes the JavaFX SDK:

```bash
mvn javadoc:javadoc \
  -DdestDir=api-reference-html \
  -DoutputDirectory=docs \
  --module-path "${JAVAFX_HOME}/lib"
```

### Output Structure

When HTML generation is enabled, the output directory includes:

```
docs/
├── api-reference.md              # Markdown API reference (always generated)
├── api-reference.json            # Structured data (always generated)
└── api-reference-html/           # Javadoc HTML site (conditional)
    ├── index.html                # Main entry point with frameset
    ├── overview-summary.html     # Package overview
    ├── allclasses-index.html     # All classes index
    ├── member-search-index.js    # Search index
    ├── com/
    │   └── example/
    │       └── app/
    │           ├── package-summary.html
    │           ├── package-tree.html
    │           └── UserService.html
    └── ...
```

### Validation

After generating HTML Javadoc, verify:

1. **`index.html` exists** in `docs/api-reference-html/`
2. **No Javadoc errors**: Check Maven output for `javadoc: error` messages — errors indicate malformed Javadoc comments that need fixing
3. **Package coverage**: All public packages from `module-info.java` (or source tree) have `package-summary.html` pages
4. **Class coverage**: All public classes have individual HTML pages
5. **Cross-links**: `{@link}` references resolve to actual HTML pages (no broken links)

### Coverage Reporting

The HTML Javadoc coverage is reported alongside the Markdown coverage in `docgen-report.json`:

```json
{
  "coverage": {
    "api": {
      "total_classes": 25,
      "classes_with_javadoc": 20,
      "coverage_percent": 80.0
    },
    "api_html": {
      "generated": true,
      "output_dir": "docs/api-reference-html/",
      "javadoc_errors": 0,
      "javadoc_warnings": 3
    }
  }
}
```

### Relationship to Markdown API Reference

The HTML and Markdown outputs serve different purposes:

| Output | Purpose | Audience |
|--------|---------|----------|
| `api-reference.md` | Quick reference in version control, IDE preview | Developers browsing in GitHub/IDE |
| `api-reference-html/` | Full searchable Javadoc site with cross-linking | Developers needing detailed API navigation |

Both outputs are generated from the same source Javadoc comments — they are different presentations of the same data. The Markdown output is always generated; the HTML output is optional (controlled by `doc_javadoc_html` config).
