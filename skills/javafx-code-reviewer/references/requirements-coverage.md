# Requirements Coverage Rules

> This document defines the check items, pass/fail criteria, and severity baselines for the **Requirements Coverage** dimension.
> Requirements coverage verifies that every requirement in the `requirements.md` specification has corresponding code implementation and test coverage, and that every code file traces back to a requirement — forming a complete bidirectional traceability chain.

## Core Principle

Requirements coverage answers two questions:
1. **Forward traceability**: Is every requirement implemented in code and covered by tests?
2. **Backward traceability**: Does every code file trace back to a legitimate requirement (no orphan code)?

A project with undocumented features or untested requirements is not ready for delivery, even if it compiles and passes runtime verification.

## Prerequisites

This dimension requires the following artifacts to exist:
- `requirements.md` — The requirements specification generated by `javafx-developer` Step 1, containing the Requirement Traceability Matrix (Section 7)
- Java source files with `@req` annotations in their Javadoc headers
- Test files with `@req` annotations and `test{Behavior}_{REQ_ID}()` naming convention

**Skip condition**: If `requirements.md` does not exist in the project, this dimension is skipped with a note: "Requirements Coverage skipped — no requirements.md found. Consider generating one with javafx-developer Step 1."

## Check Items

### Check Item 1: Requirement Implementation Coverage

**Focus**: Whether every functional requirement (FR-xxx) in `requirements.md` has at least one implementation file with a matching `@req` annotation.

**Pass Criteria**:
- Every FR-xxx ID in Section 3.1 (Feature List) has at least one Java source file annotated with `@req FR-xxx`
- Every NFR-xxx ID in Section 4 (Non-Functional Requirements) that is code-enforceable has at least one implementation file or configuration annotated with `@req NFR-xxx`
- The RTM table in Section 7.1 lists all requirement IDs with their implementation file paths

**Fail Criteria** (any one constitutes failure):
- Any FR-xxx in the feature list has no corresponding `@req` annotation in any source file
- Any requirement ID in the RTM table has an empty implementation file path
- RTM table is missing requirement IDs that exist in the feature list

**Severity Baseline**: Major

---

### Check Item 2: Requirement Test Coverage

**Focus**: Whether every functional requirement (FR-xxx) has at least one test case that validates it.

**Pass Criteria**:
- Every FR-xxx has at least one test method annotated with `@req FR-xxx` or named with the `_{REQ_ID}` suffix (e.g., `testUserCreation_FR_001()`)
- The RTM table in Section 7.1 lists test case references for all functional requirements
- Test coverage percentage (tested requirements / total requirements) >= 80%

**Fail Criteria** (any one constitutes failure):
- Any FR-xxx has no corresponding test method
- Test coverage percentage < 80%

**Severity Baseline**: Major
- De-escalation condition: Project is a prototype/MVP with explicitly deferred testing → Minor
- Escalation condition: Production application with 0% test coverage on functional requirements → Critical

---

### Check Item 3: Orphan Code Detection

**Focus**: Whether every Java source file has a valid `@req` annotation tracing back to a defined requirement.

**Pass Criteria**:
- All Java source files in `src/main/java/` have `@req` annotations in their class Javadoc header
- All `@req` annotation values reference requirement IDs that exist in `requirements.md`
- Files annotated with `@req INFRA` are legitimate infrastructure/utility files (e.g., `App.java`, database configuration, utility classes)

**Fail Criteria** (any one constitutes failure):
- Any Java source file in `src/main/java/` lacks a `@req` annotation entirely
- Any `@req` annotation references a requirement ID that does not exist in `requirements.md`
- Any file annotated with `@req INFRA` contains business logic that should be traced to a functional requirement

**Severity Baseline**: Major
- De-escalation condition: File is a generated/config file (e.g., `module-info.java`, `module-info.java`) → Minor
- Escalation condition: Multiple untracked files containing significant business logic → Critical

---

### Check Item 4: RTM Consistency

**Focus**: Whether the Requirement Traceability Matrix in `requirements.md` Section 7 is consistent with the actual code annotations.

**Pass Criteria**:
- Every `@req` annotation found in source code is listed in the RTM table
- Every entry in the RTM table has a valid file path that exists in the project
- Coverage summary numbers (Section 7.2) match the actual counts
- Untraced code list (Section 7.3) is populated if any files lack `@req` annotations

**Fail Criteria** (any one constitutes failure):
- RTM table references file paths that do not exist
- Coverage summary numbers do not match actual counts (off by more than 1 due to rounding)
- `@req` annotations in code reference requirement IDs not listed in the RTM

**Severity Baseline**: Minor
- Escalation condition: RTM is completely empty despite code having `@req` annotations → Major

---

### Check Item 5: Test Method Naming Convention

**Focus**: Whether test methods follow the `test{Behavior}_{REQ_ID}()` naming convention.

**Pass Criteria**:
- All test methods that validate a specific requirement include the `_{REQ_ID}` suffix
- The `{REQ_ID}` in the method name matches the `@req` annotation in the method's Javadoc
- Test methods that validate multiple requirements use the primary requirement ID in the name

**Fail Criteria** (any one constitutes failure):
- Test method has `@req` annotation but method name does not include the `_{REQ_ID}` suffix
- Test method name includes a `{REQ_ID}` that does not match its `@req` annotation

**Severity Baseline**: Minor

---

### Check Item 6: Non-Functional Requirement Verification

**Focus**: Whether non-functional requirements (performance, security, compatibility) have corresponding implementation evidence.

**Pass Criteria**:
- NFR-PERF-xxx requirements have corresponding code-level evidence (e.g., caching, lazy loading, background threading)
- NFR-SEC-xxx requirements have corresponding security implementations (e.g., input validation, prepared statements)
- NFR-COMPAT-xxx requirements have corresponding configuration (e.g., `--enable-native-access`, platform-specific packaging config)

**Fail Criteria** (any one constitutes failure):
- Any NFR-PERF-xxx requirement has no implementation evidence in code
- Any NFR-SEC-xxx requirement has no security implementation

**Severity Baseline**: Major
- De-escalation condition: NFR is documented as "future work" with stakeholder approval → Minor

## Annotation Format Reference

### `@req` Annotation

The `@req` tag is a custom Javadoc tag placed in the class-level or method-level Javadoc comment:

**Class-level** (links a file to the requirement(s) it implements):
```java
/**
 * User management controller handling CRUD operations.
 *
 * @req FR-001, FR-002
 * @author javafx-developer
 */
public class UserController { ... }
```

**Method-level** (links a test method to the requirement it validates):
```java
/**
 * Verifies that a new user can be created with valid input.
 *
 * @req FR-001
 */
@Test
void testUserCreation_FR_001() { ... }
```

**Infrastructure annotation** (for files not tied to a specific requirement):
```java
/**
 * Application entry point.
 *
 * @req INFRA
 */
public class App extends Application { ... }
```

### Requirement ID Naming Convention

| Prefix | Type | Source |
|--------|------|--------|
| `FR-xxx` | Functional Requirement | `requirements.md` Section 3.1 |
| `NFR-PERF-xxx` | Performance Requirement | `requirements.md` Section 4.1 |
| `NFR-SEC-xxx` | Security Requirement | `requirements.md` Section 4.2 |
| `NFR-COMPAT-xxx` | Compatibility Requirement | `requirements.md` Section 4.3 |
| `NFR-UI-xxx` | UI/UX Requirement | `requirements.md` Section 5 |
| `INFRA` | Infrastructure (no specific requirement) | N/A — utility/config files |

## Coverage Calculation

### Implementation Coverage
```
Implementation Coverage % = (Requirements with @req in code / Total requirements) × 100
```

### Test Coverage
```
Test Coverage % = (Requirements with test @req / Total functional requirements) × 100
```

### Overall RTM Coverage
```
Overall Coverage % = (Requirements with both implementation AND test / Total requirements) × 100
```

## Typical Failure Examples

**Unimplemented requirement**:
```
requirements.md lists FR-003 "Export data to CSV" but no source file has @req FR-003
→ Major: Requirement FR-003 has no implementation
```

**Orphan code**:
```
src/main/java/com/example/util/CsvExporter.java has no @req annotation
→ Major: File CsvExporter.java has no @req annotation — orphan code detected
```

**Missing test coverage**:
```
FR-001 is implemented (UserController has @req FR-001) but no test method has @req FR-001
→ Major: Requirement FR-001 has no test coverage
```

**RTM inconsistency**:
```
RTM lists FR-001 → UserController.java, but UserController.java has @req FR-002
→ Minor: RTM entry for FR-001 does not match actual @req annotation in UserController.java
```
